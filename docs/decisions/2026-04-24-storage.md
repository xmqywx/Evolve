# ADR — SQLite as single source of truth; Postgres scheduled for removal

- **Date**: 2026-04-24
- **Status**: Accepted
- **Context**: stabilization sprint, pre-multi-digital-human

## Context

`config.yaml` currently has both:

```yaml
postgres:
  dsn: "postgresql://ying@localhost/myagent"
  enabled: true
```

and SQLite as `agent.db` (see `config.yaml → agent.db_path`). Every design document (the 5 old specs + OVERVIEW.md) describes the persistence layer as SQLite. Postgres appears only as a config flag and some module-level references. Dual-enabled storage with no documented source-of-truth is a latent correctness bug: every future feature that persists data has to answer "which DB?" with no authoritative answer, and drift between the two becomes undetectable.

## Decision

**SQLite (`agent.db`) is the single source of truth for all MyAgent data.** Postgres is not a source of truth now, and is scheduled for removal in the next stabilization round.

## Rationale

1. **All existing specs describe SQLite.** The 5 old specs and the merged `docs/OVERVIEW.md` describe tables and DB operations in SQLite terms. There is no spec justifying Postgres.
2. **Single-machine personal agent.** MyAgent runs on one Mac Studio, one user. Postgres's concurrency, network access, and replication features are not needed.
3. **Operational simplicity.** SQLite has zero-admin, file-level backup, lock-free single-writer within a process. Less to debug, less to break.
4. **Reduced surface for future bugs.** Two storage paths mean every future query risks divergence. Killing one path eliminates a whole class of bugs pre-emptively.
5. **Embedding / vector search concerns do not force Postgres.** The `myagent/embedding.py` module and Doubao embedding integration do not require Postgres's pgvector — embeddings can live in a SQLite column (JSON-encoded or Float-blob) for this scale.

## Removal checklist (NOT executed this round)

This round (stabilization) only captures the decision. Execution is deferred to a follow-up.

1. **Flip the flag** — set `postgres.enabled: false` in `config.yaml`.
2. **Grep every reference** — files currently mentioning Postgres:
   - Code: `myagent/config.py`, `myagent/server.py`, `myagent/embedding.py`
   - Tests: `tests/conftest.py`, `tests/test_config.py`, `tests/test_web.py`, `tests/test_crash_recovery.py`, `tests/test_embedding.py`, `tests/test_integration_feishu.py`
   - Agent card: `agents/system-monitor.md` (will update when agents/ is restructured)
   - Docs: already being addressed by this sprint (ADR + OVERVIEW.md)
3. **Audit for Postgres-only writes** — run every feature through its data-path and confirm there is no code that writes to Postgres but not SQLite. Expected result: none. If any found, migrate that data to SQLite first.
4. **Remove the `postgres:` config block** from `config.yaml` and `config.yaml.example`.
5. **Remove `PostgresConfig` loading** from `myagent/config.py`.
6. **Remove Postgres imports / connection helpers** from `myagent/server.py` and `myagent/embedding.py`.
7. **Delete Postgres-specific tests** in `tests/` or rewrite them to use SQLite fixtures.
8. **Drop the Postgres database** — once the deployed service runs ≥7 days without the flag enabled and without issues, run `dropdb myagent` on the host. (Manual step, left to Ying.)

Each step gets its own commit.

## Alternatives considered

- **Keep Postgres as primary** — rejected. No spec justifies it; no feature requires it; single-user system does not benefit from Postgres's scale features.
- **Dual-write to both** — rejected. Doubles write code, doubles bugs, forces reconciliation logic. No upside.
- **Migrate SQLite → Postgres for "future-proofing"** — rejected as YAGNI. If a future deployment actually needs Postgres (e.g. a multi-user SaaS pivot), migration at that point is straightforward.

## Consequences

- **Immediate**: no behavior change this round. ADR documents the decision.
- **Next round**: someone (Ying or an agent iteration) runs the removal checklist. Expected effort: 1–2 hours of grep-and-delete + running tests.
- **Durable**: future specs / PRs that propose writing to Postgres are blocked by this ADR. Any such proposal must first overturn this decision.
